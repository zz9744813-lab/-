# 日程时间驱动模型

## 核心理念

日程不是工单，不需要"开始"操作。任务按时间自动进入不同阶段，用户只需要在结束时提供反馈。

## 任务生命周期

```
创建 → 未开始 → 进行中 → 已结束·待反馈 → 已完成/未完成
                                    ↓
                                 重新安排 → 未开始
```

### 阶段说明

| 阶段 | 触发条件 | 用户操作 |
|------|---------|---------|
| 未来安排 (future) | 日期在未来 | 调整时间、取消 |
| 未开始 (upcoming) | 今天，未到开始时间 | 调整时间、取消 |
| 进行中 (active) | 当前时间在 startTime 和 endTime 之间 | 完成并复盘、取消 |
| 已结束·待反馈 (ended_waiting_feedback) | 已过 endTime，无反馈 | 填写反馈、标记未完成、重新安排 |
| 已完成 (done) | 用户确认完成 | 查看复盘 |
| 未完成 (missed) | 用户标记未完成 | 重新安排、查看记录 |
| 已取消 (cancelled) | 用户取消 | 重新打开、查看原因 |

## 为什么删除"开始"按钮

- 个人日程不需要手动"开始"，到了时间自然就是进行中
- 手动开始增加了不必要的操作步骤
- 很多任务是自动进行的（如学习、锻炼），不需要告诉系统"我开始了"

## 为什么删除"延期"按钮

- "延期"是工单系统的概念，暗示任务"落后了"
- 个人日程只需要"调整时间"，不带有负面含义
- "重新安排"比"延期"更中性，更适合个人使用

## 为什么结束不等于完成

- 任务到了 endTime 只是时间到了，不代表做完了
- 用户需要主动反馈：完成了还是没完成
- 这避免了"忘记标记就自动完成"的问题
- 保留了复盘和反思的机会

## 与旧数据的兼容

- `in_progress` 状态：不再直接显示"进行中"，而是根据时间计算 phase
- `delayed` 状态：视为普通任务，按时间计算 phase
- `skipped` 状态：视为"未完成"，UI 显示"未完成/错过"
- 旧的事件类型（started, delayed, skipped）保留不删除
